# Hospital CMS — Production Runbook

## Quick Reference

| Action                  | Command                                       |
| ----------------------- | --------------------------------------------- |
| Start API               | `systemctl start hospital-api`                |
| Restart API             | `systemctl restart hospital-api`              |
| Check logs              | `journalctl -u hospital-api -f`               |
| Check agent             | `systemctl status hospital-agent`             |
| Force license re-verify | `POST /api/v1/system/license/refresh`         |
| Clear all caches        | `POST /api/v1/system/cache/clear`             |
| Revoke user sessions    | `POST /api/v1/auth/revoke-all` (as that user) |
| Verify audit chain      | `GET /api/v1/audit/integrity`                 |

## 1. Initial Deployment

### Prerequisites

- Linux server (Ubuntu 22.04 LTS recommended)
- MongoDB 7.x (replica set for production — see DEPLOYMENT.md)
- Redis 7.x
- Node.js 20.x LTS
- RSA key pair generated by the installer

### Steps

```bash
# 1. Install Node.js 20
curl -fsSL https://deb.nodesource.com/setup_20.x | sudo bash -
sudo apt-get install -y nodejs

# 2. Clone / extract the application
cd /opt && sudo tar xzf hospital-cms.tar.gz
sudo chown -R hospitalcms:hospitalcms /opt/hospital-cms

# 3. Enable Corepack and pnpm
corepack enable

# 4. Install dependencies and build
cd /opt/hospital-cms
pnpm install
pnpm build:hospital

# 5. Create environment file
sudo cp .env.hospital.example /etc/hospital-cms/hospital.env
sudo chmod 600 /etc/hospital-cms/hospital.env
sudo nano /etc/hospital-cms/hospital.env  # fill in all values

# 6. Start the API and web app in separate terminals, then run the installer once
# Terminal 1
cd apps/api && pnpm dev

# Terminal 2
cd apps/web && pnpm dev
# Open http://server-ip:3000/install and complete the wizard

# 7. Install systemd units
sudo cp deploy/hospital-api.service /etc/systemd/system/
sudo cp deploy/hospital-web.service /etc/systemd/system/
sudo cp deploy/hospital-agent.service /etc/systemd/system/
sudo systemctl daemon-reload
sudo systemctl enable --now hospital-api hospital-web hospital-agent
```

## 2. Key Management

### Instance key pair (generated by installer)

- Private key: `/etc/hospital-cms/instance.key` — **never leave the server**
- Public key: registered with the control panel during installation
- Used to sign heartbeat payloads; control panel verifies before trusting metrics

### Vendor public key (`VENDOR_PUBLIC_KEY`)

- Provided by your vendor and embedded in the `.env` file
- Used to verify: license tokens, plugin manifests, theme manifests, operational commands
- If the vendor rotates keys, they will push a `ROTATE_KEYS` command through the agent

### Key rotation procedure

1. Vendor issues new key pair and pushes `ROTATE_KEYS` command to the agent
2. Agent executes the command; new keys are distributed via the desired-state mechanism
3. Update `VENDOR_PUBLIC_KEY` in `/etc/hospital-cms/env`
4. `POST /api/v1/system/cache/clear` to bust the license verification cache
5. `POST /api/v1/system/license/rotate` to persist the new signed license token

## 3. License Management

### Check current license

```bash
curl -s -H "Authorization: Bearer $ADMIN_TOKEN" \
  http://localhost:4000/api/v1/system/license | jq .
```

The response shows both the DB record and the `verified` object confirming the RSA signature is valid.

### License expiry alerts

The system dashboard (`/system`) shows days-to-expiry with an amber warning at <30 days. The agent heartbeat response always includes the current license; if it is null, the agent logs a warning.

### Renewing a license

1. Contact your vendor to issue a new license for your instance ID
2. The vendor pushes the new signed token via the control panel
3. The agent's reconciler calls `POST /api/v1/system/license/rotate` automatically
4. Verify: check `/system` dashboard — `signatureValid: true` and new expiry date shown

### Emergency license re-verification (if cache is stuck)

```bash
curl -s -X POST -H "Authorization: Bearer $ADMIN_TOKEN" \
  http://localhost:4000/api/v1/system/license/refresh
```

## 4. Audit Log Integrity

The audit log uses a SHA-256 hash chain — every entry hashes its own data plus the previous entry's hash.

### Verify chain integrity

```bash
curl -s -H "Authorization: Bearer $ADMIN_TOKEN" \
  http://localhost:4000/api/v1/audit/integrity | jq .
```

A clean chain returns `{ verified: true, checked: N }`.

### If chain integrity fails

1. Do **not** restart or modify the database
2. Note the `firstFailedId` returned by the integrity check
3. Contact your vendor — this indicates a tampered or corrupted audit record
4. Preserve a MongoDB dump: `mongodump --uri "$MONGODB_URI" --out /backup/audit-incident-$(date +%Y%m%d)/`

## 5. Security Incidents

### Compromised SUPER_ADMIN account

```bash
# 1. Immediately revoke all sessions for the account
curl -X POST -H "Authorization: Bearer $COMPROMISED_TOKEN" \
  http://localhost:4000/api/v1/auth/revoke-all

# 2. Change the password via MongoDB directly (as a break-glass measure)
mongosh "$MONGODB_URI" --eval '
  db.users.updateOne(
    { role: "SUPER_ADMIN" },
    { $set: { isActive: false, updatedAt: new Date() } }
  );
'

# 3. Re-enable with a new password via a second SUPER_ADMIN or the installer reset
```

### Suspected plugin tampering

Plugins are RSA-signed. If a plugin install fails signature verification:

1. The registry logs a `PluginSignatureError` — check `journalctl -u hospital-api | grep PluginSignatureError`
2. The plugin is **never activated** — the host system is not affected
3. Report the plugin package to your vendor

### Unexpected network outbound connections

The management agent only connects to the control-panel URL defined in `CONTROL_PANEL_URL`. All outbound connections are logged by pino at the info level. The package installer enforces an allowlist via `VENDOR_CDN_HOSTS`.

## 6. Backups

### MongoDB backup (daily, recommended)

```bash
#!/bin/bash
# /etc/cron.daily/hospital-cms-backup
set -euo pipefail
DATE=$(date +%Y%m%d-%H%M)
DEST="/backup/mongodb/$DATE"
mongodump --uri "$MONGODB_URI" --out "$DEST"
# Retain 30 days
find /backup/mongodb -maxdepth 1 -type d -mtime +30 -exec rm -rf {} +
```

### Critical collections to back up

- `users` — accounts and roles
- `patients` — patient records
- `encounters` — clinical encounters
- `audit_logs` — immutable audit chain (never delete; back up separately)
- `licenses` — license records
- `hospital_instance` — instance identity and config

### Point-in-time recovery (with replica set oplog)

```bash
mongorestore --uri "$MONGODB_URI" \
  --oplogReplay \
  --oplogLimit "$(date +%s):0" \
  /backup/mongodb/YYYYMMDD-HHMM/
```

## 7. Monitoring & Alerting

### Health check endpoints

```
GET /health        # liveness (always 200 if process is up)
GET /health/ready  # readiness (checks MongoDB + Redis)
GET /health/live   # same as /health
```

Configure your load balancer / uptime monitor to hit `/health/ready`.

### Metrics endpoint (requires SYSTEM_SETTINGS_READ)

```bash
curl -s -H "Authorization: Bearer $ADMIN_TOKEN" \
  http://localhost:4000/api/v1/system/metrics | jq .
```

### Key alert thresholds

| Metric                 | Warning | Critical    |
| ---------------------- | ------- | ----------- |
| Disk %                 | 70%     | 85%         |
| Memory %               | 75%     | 90%         |
| License days remaining | 30      | 7           |
| Heartbeat last seen    | >5 min  | >15 min     |
| Audit chain integrity  | —       | any failure |

## 8. Updates

### Application update procedure

```bash
# 1. Notify users (active encounters should be completed first)
# 2. Put API into maintenance mode (optional — remove from load balancer)
# 3. Pull new release
cd /opt/hospital-cms && pnpm install
# 4. Build
pnpm build
# 5. Restart
sudo systemctl restart hospital-api hospital-agent
# 6. Verify health
curl http://localhost:4000/health/ready
# 7. Check logs for errors
journalctl -u hospital-api --since "1 minute ago"
```

### Rolling back

```bash
# If the new version fails to start:
sudo systemctl stop hospital-api
cd /opt && sudo tar xzf hospital-cms-PREVIOUS.tar.gz -C /opt/hospital-cms
sudo systemctl start hospital-api
```

## 9. Systemd Unit Files

### `/etc/systemd/system/hospital-api.service`

```ini
[Unit]
Description=Hospital CMS API
After=network.target mongod.service redis.service
Requires=mongod.service redis.service

[Service]
Type=simple
User=hospitalcms
WorkingDirectory=/opt/hospital-cms/apps/api
EnvironmentFile=/etc/hospital-cms/env
ExecStart=/usr/bin/node dist/server.js
Restart=on-failure
RestartSec=10
StandardOutput=journal
StandardError=journal
SyslogIdentifier=hospital-api

# Security hardening
NoNewPrivileges=true
ProtectSystem=strict
ProtectHome=true
PrivateTmp=true
CapabilityBoundingSet=
AmbientCapabilities=

[Install]
WantedBy=multi-user.target
```

### `/etc/systemd/system/hospital-agent.service`

```ini
[Unit]
Description=Hospital CMS Management Agent
After=network.target hospital-api.service
Wants=hospital-api.service

[Service]
Type=simple
User=hospitalcms
WorkingDirectory=/opt/hospital-cms/apps/agent
EnvironmentFile=/etc/hospital-cms/env
ExecStart=/usr/bin/node dist/main.js
Restart=on-failure
RestartSec=30
StandardOutput=journal
StandardError=journal
SyslogIdentifier=hospital-agent

NoNewPrivileges=true
ProtectSystem=strict
ReadWritePaths=/var/lib/hospital-agent
PrivateTmp=true

[Install]
WantedBy=multi-user.target
```
